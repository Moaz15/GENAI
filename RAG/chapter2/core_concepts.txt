Retrieval-Augmented Generation (RAG) is a class of architectures that integrate explicit information retrieval into the generation process of language models.
Unlike traditional LLM inference, where responses are generated solely from parametric knowledge encoded in model weights, RAG conditions generation on externally retrieved evidence.
Formally, a standard language model estimates the conditional distribution:  𝑃(𝑦∣𝑞)
q is the input query
y is the generated output sequence

In contrast, RAG introduces a retrieval step over an external knowledge corpus 𝐶 , producing a subset of documents 𝐷={𝑑1,…,𝑑𝑘} and models:  𝑃(𝑦∣𝑞,D)

This distinction is foundational. It transforms generation from an implicit recall problem into an explicit evidence-conditioned reasoning task.

Non-Parametric Memory as a First-Class Component : 
A key conceptual shift introduced by RAG is the treatment of external knowledge as non-parametric memory.

Parametric vs Non-Parametric Memory

Parametric memory
Stored in model parameters
Learned during training
Fixed at inference time
Opaque and difficult to audit

Non-parametric memory
Stored externally (documents, databases, APIs)
Retrieved dynamically
Updateable without retraining
Explicit and inspectable

RAG systems explicitly acknowledge that knowledge is too large, too dynamic, and too domain-specific to be stored entirely in model parameters.
This separation mirrors classical computer systems:
LLM → CPU (reasoning)
Retrieval system → RAM / disk (memory)

High-Level RAG Architecture

At a systems level, RAG decomposes inference into three modular subsystems:
Knowledge Source
Retriever
Generator

Inference Pipeline

User submits query : q
Query is encoded into a vector representation :  𝑞∈𝑅𝑑
Retriever selects top-𝑘 documents from corpus C
Retrieved documents are injected into the LLM context
Generator produces output y

This modularity is not an implementation detail — it is the core design principle that enables RAG to scale.

Knowledge Sources and Corpus Design

The knowledge corpus C is the factual backbone of a RAG system

Types of Knowledge
Unstructured: PDFs, Docs, HTML, transcripts
Semi-structured: JSON, logs, events
Structured: SQL tables, feature stores, knowledge graphs

Each type introduces different constraints on:
Retrieval strategy
Chunking
Latency
Freshness guarantees

Chunking as a Modeling Decision
Documents are rarely retrieved as atomic units. Instead, they are decomposed into chunks 𝑐𝑖 to balance:
Semantic coherence
Retrieval granularity
Context window efficiency
Chunking implicitly defines the retrieval resolution of the system.

Approximate Nearest Neighbor (ANN) Retrieval

Exact nearest neighbor search is computationally infeasible for large corpora.
RAG systems rely on ANN indexes (e.g., HNSW) that trade:
Recall
Latency
Memory

This introduces retrieval stochasticity, meaning:
The retriever itself is an approximate model
Retrieval errors propagate downstream to generation

This is critical for production debugging.
